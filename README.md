- PingFederate
- ADFS
- Okta
- Custom SAML providers

The goal is to determine:

- Who initiated authentication
- Whether Entra ID redirected the user to the external IdP
- Whether the external IdP authenticated the user
- Whether the IdP returned a SAML assertion to Entra ID
- Whether Entra ID accepted or rejected the assertion
- Where the authentication flow failed

Most HAR traces contain hundreds of requests. In reality, only a few requests are relevant for authentication troubleshooting.

# Direct Federation Authentication Flow

In Direct Federation, Microsoft Entra ID does **not** authenticate the external user's password.

Instead:

1. Entra ID sends a SAML Authentication Request to the partner IdP.
2. The partner IdP authenticates the user.
3. The partner IdP sends a signed SAML assertion back to Entra ID.
4. Entra ID validates the assertion.
5. The user is redirected to guest redemption or the target application.

```text
User
 |
 v
Microsoft Entra ID (Resource Tenant)
 |
 | SAML AuthnRequest
 v
External SAML Identity Provider
(Ping / ADFS / Okta)
 |
 | User Authentication
 v
External SAML Identity Provider
 |
 | SAMLResponse
 v
Microsoft Entra ID
 |
 v
Guest Redemption / Application Access
```

# How to Read a HAR Trace

Do NOT read every request sequentially.

Focus on:

- Redirects (302, 303, 307)
- SAMLRequest
- SAMLResponse
- RelayState
- login.microsoftonline.com
- External IdP endpoints

Ignore:

```text
.css
.js
.png
.jpg
.jpeg
.svg
.woff
favicon.ico
```

Examples:

```text
main.css
jquery.js
bootstrap.js
login.js
logo.png
background.jpg
```

These are usually page-rendering assets and not part of the authentication flow.

# Key Authentication Terms

## SAMLRequest

A SAML Authentication Request generated by Entra ID and sent to the external IdP.

Example:

```text
SAMLRequest=
```

Meaning:

```text
Entra ID → Please authenticate this user.
```

---

## SAMLResponse

A signed SAML assertion returned by the external IdP after authentication.

Example:

```text
SAMLResponse=
```

Meaning:

```text
External IdP → User authenticated successfully.
```

---

## RelayState

A state-tracking value used throughout the federation transaction.

Example:

```text
RelayState=xyz123
```

RelayState commonly contains:

- Tenant context
- Application context
- Redirect information
- Session state
- Federation transaction state

### Important

If Entra sends:

```text
SAMLRequest
RelayState=abc123
```

The external IdP must return:

```text
SAMLResponse
RelayState=abc123
```

The value must be preserved exactly.

Failure to return RelayState can cause Entra ID to lose context about the original authentication request.

# Common Microsoft Entra ID URLs

When reviewing a HAR trace, these URLs usually indicate Microsoft Entra ID involvement:

```text
login.microsoftonline.com
aadcdn.msftauth.net
login.live.com
```

Common federation endpoint:

```text
https://login.microsoftonline.com/login.srf
```

Example:

```text
POST https://login.microsoftonline.com/login.srf
```

Meaning:

```text
External IdP
      |
      | SAMLResponse
      v
Microsoft Entra ID
```

At this step, Entra ID is processing the federated SAML assertion.
